# 文档生成提示词

本文档提供生成项目文档的标准化提示词模板，适用于各类代码项目的文档生成。

## 项目概述生成提示词

```
根据代码帮我完成项目概述输出到doc/summary.md

要求：
- 不使用 H1、H2 标签
- 不需要项目概述标题
- 不需要依赖项说明
- 适当描述项目特点，吸引用户使用
- 使用 Markdown 格式
```

### 项目概述内容要点

- **核心功能**：简要说明项目的主要功能
- **技术特点**：突出项目的技术亮点
- **应用场景**：描述适用的使用场景
- **优势价值**：说明相比其他方案的优势

### 示例结构

```markdown
[项目名称] 是一个专为 [目标领域] 设计的 [核心功能] 解决方案。它基于 [技术栈] 构建，提供了 [主要特性] 能力，帮助开发者轻松实现 [目标功能]。

## 核心特性

- 特性1：描述
- 特性2：描述
- 特性3：描述

## 应用场景

适用于以下场景：
- 场景1
- 场景2

## 技术优势

- 优势1
- 优势2
```

## API 文档生成提示词

```
根据代码帮我完成 API 文档输出到doc/api.md

要求：
- 不使用 H1、H2 标签
- 不需要 API 文档标题
- 使用 H3 及之后级别的标题
- API 部分优先使用 table 格式
- API 部分不需要示例代码
- 严格遵守格式要求
```

### API 文档内容要点

#### 配置选项
使用表格列出所有配置项：
- 参数名称
- 类型
- 默认值
- 说明

#### 方法/函数
使用表格列出所有公开方法/函数：
- 方法名称
- 参数
- 返回值
- 说明

#### 装饰器/属性
列出所有公开的装饰器或属性：
- 名称
- 类型
- 说明

### 示例结构

```markdown
### 配置选项

| 参数名 | 类型 | 默认值 | 说明 |
|--------|------|--------|------|
| optionName | string | 'default' | 选项说明 |
| optionName2 | number | 100 | 选项说明 |

### 方法

| 方法名 | 参数 | 返回值 | 说明 |
|--------|------|--------|------|
| methodName | (param1, param2) | Promise<Result> | 方法说明 |
| methodName2 | (config) | void | 方法说明 |

### 装饰器

| 装饰器名 | 类型 | 说明 |
|----------|------|------|
| decoratorName | Function | 装饰器说明 |
```

## 通用文档生成规范

### Markdown 层级规则
- 不使用 H1 (`#`)
- 不使用 H2 (`##`)
- 使用 H3 (`###`) 作为顶级标题
- 使用 H4 (`####`) 作为次级标题
- 避免过深的嵌套层级

### 表格格式规范
- 使用标准 Markdown 表格语法
- 表头包含：名称、类型、默认值/参数、说明
- 对齐方式：左对齐即可
- 保持列宽合理，便于阅读

### 内容组织原则
- 配置选项按逻辑分组
- 相关方法归类展示
- 保持章节间逻辑连贯
- 避免重复说明

### 示例代码处理
- 配置示例：放在具体配置项后
- 使用方法示例：放在方法表格后（如有需要）
- 保持示例简洁明了

## 自定义模板

根据项目类型，可以自定义以下部分：

### 库/包类型项目
```
根据代码帮我完成项目概述输出到doc/summary.md，API文档输出到doc/api.md
markdown不要使用h1、h2对应的标签
doc/summary.md不需要项目概述标题，不需要依赖项说明
doc/api.md不需要API文档标题，使用h3及之后
优先使用table格式展示API，API部分不需要示例代码
```

### 框架/插件类型项目
```
根据代码帮我完成项目概述输出到doc/summary.md，API文档输出到doc/api.md
重点描述框架特性和插件机制
markdown不要使用h1、h2对应的标签
优先使用table格式展示配置和API
```

### 应用类型项目
```
根据代码帮我完成项目概述输出到doc/summary.md，API文档输出到doc/api.md
重点描述应用功能和使用场景
markdown不要使用h1、h2对应的标签
优先使用table格式展示API和配置
```
